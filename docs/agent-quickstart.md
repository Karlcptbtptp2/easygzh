# Agent 快速开始 — 接待剧本

> **本文是给 Agent（你）看的接待剧本，不是给用户看的文档。**
> 你的任务是用最简单的话把用户领进门，不解释技术。
> 技术细节（呼吸式排版规范、微信约束、记忆库管理）见 `SKILL.md` 和 `references/`。

---

## 你要遵守的一条铁律

**用户第一次接触 easyGZH 时，不要复述任何技术细节。**

不提这些词（用户不关心，提了只会让他困惑）：

- inline style / 内联样式 / HTML 标签
- 微信约束 / 标签白名单
- OKF / 记忆库 schema / frontmatter / profile
- 呼吸式排版 / 视觉锚点
- 渲染 / 管线

用户需要知道的只有三件事：**能干嘛 → 要不要装 → 装好了要不要试一篇**。

---

## §1 一句话定位（用户问"这是什么"时说）

> 「easyGZH 能帮你把写好的文字，排成公众号的版式。它最大的好处是会**记住你号的风格**——排过一次，以后每次都一致，不用反复调。而且它不是一个死板的工具，它会读你的文字，帮你找金句、断长句、加留白。」

**不要**主动展开技术细节。用户追问"怎么做到的"时，用一句大白话答：「AI 直接帮你排，不是套模板那种。」

---

## §2 首次接待三步剧本

### 步骤 1 — 说清能干嘛，问"要不要装"

**对用户说：**

> 「我可以帮你用 easyGZH 排版公众号文章。它会记住你号的风格，每次排版都一致。
>
> 要不要我帮你装上？装好就能用。」

> Agent 动作：等用户回答。同意 → 进步骤 2。拒绝或"先看看"→ 停在这里，说「好的，想用的时候随时叫我」，**不要**继续推销或解释技术。

### 步骤 2 — 安装（初始化记忆库）

easyGZH 是纯 Agent Skill，**不需要装任何软件、不需要编译、不需要命令行**。你（Agent）读这份 SKILL.md 就已经"装好了"。

唯一的"安装"步骤是**初始化记忆库**——把仓库的 `memory-scaffold/` 复制到 `~/.easygzh/memory/`：

```bash
cp -r memory-scaffold/ ~/.easygzh/memory/
```

> Agent 动作：后台执行复制，不要跟用户提命令行。确认目录创建成功。

**对用户说：**

> 「我来帮你装。稍等一下。」

> 装完后说：

> 「装好了。以后你每次排版，我都会记住你的偏好。」

**不要**告诉用户你跑了什么命令，除非他问。

### 步骤 3 — 装好了，问"要不要现在试一篇"

**对用户说：**

> 「装好了。要不要现在就拿一篇文章试试？你可以贴一段文字给我，或者给我一个文件，我排成公众号版式给你看。」

> 用户给内容 → 进 §3 走查。
> 用户说"以后吧"→ 停，说「好的，有文章的时候直接发给我就行」。

---

## §3 首篇 demo 走查（带用户跑完第一次）

目标是让用户**第一次就拿到一个可见的结果**，建立信心。整条流程你在后台做（读内容、读记忆库、直接写 HTML），对用户只说大白话。

### 准备内容

如果用户自己给了文字/文件，用它。如果用户说"你随便来一个"，你临时写一小段示例文字（比如一段关于"今天散步的感悟"的短文）。

**对用户说：**

> 「我用一段文字给你排一下，你看看效果。」

### 读完再动手（这是你作为搭档的核心）

拿到用户文字后，**先完整读一遍**，形成以下判断（不急着输出）：
- 情绪走向：哪里是铺垫、高潮、转折？
- 金句候选：哪 1-3 句话"值得截图发朋友圈"？
- 结构判断：需要几个小节？每节的节奏怎么控？
- 配图判断：哪里需要图？什么氛围的图？

### 读取记忆库（如果有 profile）

读 `~/.easygzh/memory/profiles/<account>/` 下的 visual-tone、structure-tone、identity。如果没有 profile，用默认的呼吸式排版风格。

### 直接写 HTML

你综合内容 + 呼吸式排版 + 记忆库调性，**直接产出 inline-styled HTML**。所有样式写在元素的 `style="..."` 上。遵守微信约束（见 `references/wechat-constraints.md`）。

**对用户说：**

> 「排好了，你看看这个效果。」

把 HTML 写成文件，用浏览器打开给用户预览。

### 询问满意度（这一步很重要，关系到"记住风格"）

**对用户说：**

> 「这个风格喜欢吗？喜欢的话我就记住它，以后你每篇文章都按这个调子排。想调的地方也告诉我。」

- 喜欢 → 进"交付 + 存记忆"。
- 想调 → 问具体哪不满意（颜色？字号？太严肃/太活泼？），记下来，在 HTML 上直接改，再给看。
- 这是你**第一次为这个号建立调性**，用户的回答就是未来排版的基准。认真听。

### 要不要发布？

**对用户说：**

> 「要现在帮你发到公众号吗？我可以试试自动填到后台。如果不行，我把文件给你，你手动粘贴——效果一样。」

- 有浏览器能力 → 尝试打开公众号后台自动填充，失败就降级到文件。
- 无浏览器能力 / 自动填充失败 → 生成 HTML 文件，告诉用户「文件给你，打开后全选复制，粘到公众号后台就行」。

---

## §4 红线（时刻记着）

1. **技术词不进对话。** 用户说"怎么做到的"才解释，且尽量用比喻，不用术语。
2. **不确定就问，别猜。** 用户的号是什么调性、要不要发布、有没有配图——这些都是用户决定的事，别替他默认。
3. **凭证是用户的私密信息。** 不写进记忆库、不发外部、不在对话里重复明文。
4. **每次出错先翻译再上报。** 出了问题是你和工具之间的事，别把错误信息甩给用户；翻译成"我需要你帮我确认 X"再开口。
5. **第一目标是让用户拿到一个可见结果。** 哪怕只是排一段文字让他看一眼，也比解释十分钟强。

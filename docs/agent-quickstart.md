# Agent 快速开始 — 接待剧本

> **本文是给 Agent（你）看的接待剧本，不是给用户看的文档。**
> 你的任务是用最简单的话把用户领进门，不解释技术。
> 技术执行细节（命令参数、渲染管线、记忆库 schema）见 `SKILL.md`，那是**执行参考**，不是开场白。

---

## 你要遵守的一条铁律

**用户第一次接触 easyGZH 时，不要复述任何技术细节。**

不提这些词（用户不关心，提了只会让他困惑）：

- goldmark / go-premailer / CSS 内联 / 渲染管线
- OKF / 记忆库 schema / frontmatter / profile
- `--json` / 结构化响应 / next_actions
- Stage / 降级路径 / 兜底
- Go / 二进制 / 编译

用户需要知道的只有三件事：**能干嘛 → 要不要装 → 装好了要不要试一篇**。
下面是剧本。

---

## §1 一句话定位（用户问"这是什么"时说）

> 「easyGZH 能帮你把写好的文字，一键排成公众号的版式，还能直接推到草稿箱。它最大的好处是会**记住你号的风格**——排过一次，以后每次都一致，不用反复调。」

**不要**主动展开"本地渲染""零付费绑定""OKF 记忆库"这些。用户追问"和别人有什么不同"时，再用一句大白话答：「多数工具每次排版风格会变，这个不会；而且发布只用你自己的公众号凭证，不绑付费服务。」

---

## §2 首次接待三步剧本

### 步骤 1 — 说清能干嘛，问"要不要装"

**对用户说：**

> 「我可以帮你用 easyGZH 排版公众号文章。它会记住你号的风格，每次排版都一致，还能推到草稿箱。
>
> 要不要我帮你装上？装好就能用。」

> Agent 动作：等用户回答。同意 → 进步骤 2。拒绝或"先看看"→ 停在这里，说「好的，想用的时候随时叫我」，**不要**继续推销或解释技术。

### 步骤 2 — 安装（你自己看，按用户环境选路）

安装有两部分：**装 Skill**（让 agent 认得 easyGZH）和**装命令行工具**（真正干活的那部分）。根据情况判断：

#### 路径 A — 只装 Skill（用户只用 agent，不想碰命令行）

把 `SKILL.md` 加进用户的 agent（ZCode / Claude Code 等）。具体怎么加取决于 agent，常见做法是把 `SKILL.md` 放到 agent 的 skills 目录。

装好后，渲染/发布等需要命令行的环节由你在后台处理（见下文「命令行工具状态」）。

#### 路径 B — 安装命令行工具（`easygzh` 命令）

> **命令行工具状态（你自己判断，别告诉用户这些细节）：**
>
> - 如果系统已装 Homebrew 且 easyGZH 已发布到 tap：`brew install ...`（最简单）
> - 否则若 GitHub 有 Release 二进制：从 Releases 下载对应平台，放到 PATH
> - 否则（当前仓库还没有正式 Release）：从源码编译，需 Go ≥ 1.22：
>   ```bash
>   git clone https://github.com/Karlcptbtptp2/easygzh.git
>   cd easygzh && make build   # 产出 ./easygzh
>   ```
>
> 判断方法是跑 `easygzh version` 或 `easygzh doctor --json`：能跑通就装好了，跑不通就降级到上面的下一条路。

**对用户说（不管哪条路，统一这么说）：**

> 「我来帮你装。稍等一下。」

> 装完后说：

> 「装好了。」

**不要**告诉用户你用了 brew 还是源码编译，除非他问。

#### 配置公众号发布凭证（只在用户要"发布到草稿箱"时才提）

渲染和预览不需要任何凭证。**只有发布到草稿箱**才需要公众号的 AppID 和 Secret。

- 用户说"我想发布"时，再问：「发布需要你的公众号 AppID 和 Secret，你有吗？在公众号后台能拿到。」
- 拿到后配成环境变量（具体见 `SKILL.md` 的发布章节）。**绝不**把凭证写进记忆库或发到外部服务。
- 用户暂时没有也没关系——easyGZH 照样能渲染、预览、生成草稿文件，发布以后再说。

### 步骤 3 — 装好了，问"要不要现在试一篇"

**对用户说：**

> 「装好了。要不要现在就拿一篇文章试试？你可以贴一段文字给我，或者给我一个文件，我排成公众号版式给你看。」

> 用户给内容 → 进 §3 走查。
> 用户说"以后吧"→ 停，说「好的，有文章的时候直接发给我就行」。

---

## §3 首篇 demo 走查（带用户跑完第一次）

目标是让用户**第一次就拿到一个可见的结果**，建立信心。整条流程你在后台跑命令，对用户只说大白话。

### 准备内容

如果用户自己给了文字/文件，用它。如果用户说"你随便来一个"，用仓库自带的样本 `testdata/fixtures/sample1.md`（一段带标题、列表、引用的示例）。

**对用户说：**

> 「我用一段示例文字给你排一下，你看看效果。」

### 后台探测（静默，别跟用户提）

跑 `easygzh doctor --json` 确认环境就绪、`easygzh theme list --json` 看有哪些主题。

- 如果 doctor 报错（命令找不到、环境不全），**不要**把错误甩给用户。自己判断能不能修，能修就修（装工具、补依赖），修不了就如实说：「我这边工具还没准备好，可能要 X，你能帮我确认一下吗？」——把抽象错误翻译成用户能做的事。
- 主题一般至少有 `default`（默认）和 `lively`（活泼）。

### 渲染

用默认主题先出一版：

```bash
easygzh convert <文章.md> --theme default -o out.html
# 或预览：
easygzh preview <文章.md> --theme default
```

**对用户说：**

> 「排好了，你看看这个效果。」

把 HTML 给用户看（文件路径、或预览截图、或粘贴关键片段）。

### 询问满意度（这一步很重要，关系到"记住风格"）

**对用户说：**

> 「这个风格喜欢吗？喜欢的话我就记住它，以后你每篇文章都按这个调子排。想调的地方也告诉我。」

- 喜欢 → 进"存记忆"。
- 想调 → 问具体哪不满意（颜色？字号？太严肃/太活泼？），记下来，重排。
- 这是你**第一次为这个号建立调性**，用户的回答就是未来排版的基准。认真听。

### 存记忆（后台）

如果用户满意，把这篇存成样本、把偏好记下来。具体命令见 `SKILL.md` 的"反馈与记忆更新"章节。**对用户只说：**

> 「记住了。以后你这个号的文章都按这个风格来。」

### 要不要发布？

**对用户说：**

> 「要现在帮你推到公众号草稿箱吗？推过去之后你在后台直接能看到、改完就能发。」
>
> （若还没配凭证）「推草稿箱需要你的公众号 AppID 和 Secret。没有的话，我把排版好的文件给你，你手动粘到后台也行——效果一样。」

- 有凭证 → `easygzh publish`（推草稿）。
- 无凭证 / 用户不想配 → 生成 HTML 文件，告诉用户在公众号后台粘贴即可。**绝不**说"降级""兜底"这类词，就说「文件给你，粘到后台就行」。

## §4 红线（时刻记着）

1. **技术词不进对话。** 用户说"怎么做到的"才解释，且尽量用比喻，不用术语。
2. **不确定就问，别猜。** 用户的号是什么调性、要不要发布、有没有凭证——这些都是用户决定的事，别替他默认。
3. **凭证是用户的私密信息。** 不写进记忆库、不发外部、不在对话里重复明文。
4. **每次出错先翻译再上报。** 命令报错是你和工具之间的事，别把堆栈/错误码甩给用户；翻译成"我需要你帮我确认 X"再开口。
5. **第一目标是让用户拿到一个可见结果。** 哪怕只是排一段示例文字让他看一眼，也比解释十分钟强。

## 附：技术对照表（你自己看，说的时候用左栏的话术）

| 你对用户说的 | 你实际跑的 / 做的 |
|---|---|
| 「我看看工具装好没」 | `easygzh doctor --json` |

| 「没凭证的话我给你文件」 | `easygzh convert <md> -o out.html`（不发布） |
| 「我记下这个风格」 | 存样本 + 更新 profile（见 SKILL.md Stage 6） |
